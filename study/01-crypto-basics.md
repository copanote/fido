# 01. 암호화 기초 — FIDO를 이해하기 위한 최소한의 암호학

FIDO는 전적으로 **공개키 암호화**와 **디지털 서명** 위에서 동작합니다.
이 두 개념을 제대로 이해하면 FIDO의 모든 흐름이 자연스럽게 이해됩니다.

---

## 1. 대칭키 암호화 (Symmetric Key) — 과거의 방식

가장 직관적인 암호화. 잠그는 키와 여는 키가 같다.

```
평문  --[같은 키로 암호화]--> 암호문  --[같은 키로 복호화]--> 평문
```

**문제점:**
- 서버와 클라이언트가 같은 키를 공유해야 한다.
- 키를 어떻게 안전하게 전달할 것인가? → **키 배송 문제**
- 서버가 해킹되면 키가 통째로 유출된다.

비밀번호를 서버에 저장하는 것도 이 범주. 서버 DB가 털리면 모든 사용자 비밀번호가 위험.

---

## 2. 공개키 암호화 (Asymmetric Key) — FIDO의 핵심

잠그는 키(공개키)와 여는 키(개인키)가 **다르다.**

```
키 쌍 = { 공개키(Public Key), 개인키(Private Key) }

공개키로 암호화 → 개인키로만 복호화 가능
개인키로 서명   → 공개키로만 검증 가능
```

가장 널리 쓰이는 알고리즘: **RSA**, **ECDSA (타원곡선)**
FIDO에서는 주로 **ECDSA (ES256)** 를 사용한다.

### 핵심 성질

1. 공개키를 알아도 개인키를 알 수 없다 (수학적으로 역산 불가)
2. 공개키는 누구에게나 공개해도 안전하다
3. 개인키 소유자만이 서명을 생성할 수 있다

---

## 3. 디지털 서명 (Digital Signature) — FIDO 인증의 본질

디지털 서명은 이 질문에 답한다:
> "이 데이터가 정말로 그 사람(기기)에서 온 것인가?"

### 서명 생성

```
서명 = Sign(개인키, 데이터)
```

- 개인키를 가진 사람만 서명을 만들 수 있다
- 같은 데이터라도 매번 서명 값이 다를 수 있다 (랜덤 요소 포함)

### 서명 검증

```
결과 = Verify(공개키, 데이터, 서명)
→ true: 개인키 소유자가 이 데이터에 서명했음이 확실
→ false: 위조되었거나, 데이터가 변조됨
```

### 실생활 비유

```
도장 비유:
  - 개인키 = 나만 가진 도장
  - 공개키 = 내 도장 모양을 담은 공개 도감
  - 서명   = 도장을 찍은 자국
  - 검증   = 도감과 대조하여 진짜 도장인지 확인
```

## 4. 챌린지-응답 (Challenge-Response) — 리플레이 공격 방지

서명만으로는 부족하다. 공격자가 이전에 캡처한 서명을 재사용할 수 있기 때문이다.

**리플레이 공격:**
```
1. 공격자가 정상 인증 패킷을 캡처: { 서명: "ABCD..." }
2. 나중에 같은 패킷을 서버에 재전송
3. 서명이 유효하므로 서버가 인증 허용 → 공격 성공
```

**챌린지-응답으로 해결:**
```
서버 → 클라이언트: "이번 챌린지: a3f9b2c1 (매번 새로운 무작위값)"
클라이언트 → 서버: "서명(개인키, 챌린지값)"
서버: 검증(공개키, 챌린지값, 서명) → 성공
```

챌린지는 한 번 사용 후 폐기. 공격자가 서명을 캡처해도 그 챌린지는 이미 만료.

## 5. 해시 함수 (Hash Function)

서명을 만들 때 데이터 전체를 서명하지 않고, 해시값을 서명한다.

```
데이터 --[해시 함수(SHA-256)]--> 고정 길이 해시값 (32바이트)
           → 해시값만 서명 (훨씬 빠름)
```

**해시 함수의 성질:**
- 같은 입력 → 항상 같은 출력
- 다른 입력 → 완전히 다른 출력 (눈사태 효과)
- 해시값에서 원본 데이터 복원 불가 (단방향)
- 충돌 저항성: 같은 해시값을 가진 다른 데이터를 찾기 사실상 불가능

## 6. 인증서 (Certificate) — Attestation의 기초

Attestation(인증기 검증)을 이해하려면 인증서 개념이 필요하다.

```
인증서 = { 공개키 + 소유자 정보 + 유효기간 + CA의 서명 }
```

- **CA (Certificate Authority):** 인증서의 신뢰를 보증하는 기관
- CA가 서명한 인증서 → "이 공개키는 정말 이 기기의 것"임을 증명

**FIDO Attestation에서의 역할:**
```
기기 제조사 (삼성, 구글 등)
  ↓ 서명
기기에 심어진 Attestation 인증서
  ↓ 포함
기기가 FIDO 등록 시 서버에 제출
  ↓ 서버가 검증
"이 기기는 FIDO Alliance 인증을 받은 정품 기기입니다"
```

## 7. FIDO에서 실제로 쓰이는 알고리즘

| 용도 | 알고리즘 | 설명 |
|------|---------|------|
| 키 쌍 생성 | ECDSA P-256 | ES256, FIDO2에서 기본 |
| 해시 | SHA-256 | 256비트 해시 |
| 키 쌍 생성 (대안) | RSA-PKCS1v1.5, RS256 | 일부 구형 기기 |
| 난수 생성 | CSPRNG | 챌린지 생성에 사용 |

### ECDSA P-256 키 예시 (실제 형태)

```
개인키 (32바이트, 절대 공개 불가):
  d4e1... (256비트 정수)

공개키 (64바이트, 서버 저장):
  x: 8f2a... (32바이트)
  y: 3c9d... (32바이트)
  → (x, y) 좌표로 표현되는 타원곡선 위의 점
```

## 8. 정리: FIDO 인증의 수학적 본질

```
[등록]
1. 기기에서 (개인키 d, 공개키 Q) 생성
2. 서버에 Q 저장
3. d는 기기 TEE에 보관

[인증]
1. 서버 → 기기: challenge c (무작위)
2. 기기: signature σ = ECDSA_Sign(d, SHA256(c || clientData))
3. 기기 → 서버: σ
4. 서버: ECDSA_Verify(Q, SHA256(c || clientData), σ) == true?
```

이 4단계가 모든 FIDO 인증의 수학적 핵심입니다.
UAF든 FIDO2든 Passkey든, 결국 이 구조 위에 편의성과 보안 레이어를 추가한 것입니다.
